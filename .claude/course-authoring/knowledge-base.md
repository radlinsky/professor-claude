# Knowledge base contract

Goal: a paper-agnostic, citable encyclopedia of everything the repo's sources
teach — definitions, notation, derivations, terminology, misconceptions, teaching
insights, prerequisite structure — that grows as sources are ingested
(`extract-knowledge` skill) and feeds course building (`create-course` Phase 1),
gap analysis (`knowledge-gap-check`), and auditing (checks 22–23).

## Layout

```
knowledge/
├── README.md          # index; table bodies GENERATED by scripts/gen-kb-index.R
├── references.bib     # the one bibliography (see citations.md)
├── glossary.md        # tier 2: atomic terms, one cited table row each
├── concepts/<slug>.md # tier 1: one page per lesson-anchoring concept
└── sources/<slug>.md  # one record per ingested source (bibliography, license,
                       #   big ideas, chapter coverage map = the resume state)
```

Templates: `kb-concept-template.md` and `kb-source-template.md` in this directory —
copy them, never improvise the section structure. Pages are plain `.md` with a
YAML `title:` line only (they render on the site; Quarto's built-in search makes
the KB browsable). After creating, renaming, or deleting any KB page, run
`Rscript scripts/gen-kb-index.R` from the repo root; `--check` is the CI drift
guard. Never hand-edit inside its sentinel blocks.

## What is a concept? (granularity heuristic)

A concept gets its own `knowledge/concepts/<slug>.md` page only if it passes BOTH
tests:

1. **Independently citable** — sources make claims about it on its own, not only
   as a step inside something else.
2. **Lesson anchor** — it could plausibly anchor ONE foundations lesson (the same
   genericity instinct as `course-structure.md`'s foundations rule).

Consequences:

- Two ideas always taught together and never cited separately → **one page**
  (e.g. "odds and log-odds", not two).
- A big topic is SEVERAL pages, not one (e.g. "game theory" decomposes into
  payoff-matrix, Nash-equilibrium, … — each independently citable and
  lesson-sized).
- A page whose *Key results* section grows past ~2 screens is telling you it is
  two concepts — split it and cross-link.
- **Fails test 2 → glossary row, not a page** (tier 2, below).

Slugs: kebab-case. If the concept maps 1:1 to an existing `foundations/<slug>`
module, use the SAME slug. Check `knowledge/README.md` and `knowledge/concepts/`
before minting a slug — a collision means merge into the existing page, never a
duplicate.

## Tier 2: the glossary

Atomic vocabulary — terms with a definition but no lesson's worth of structure
("estimand", "iid", "GDP") — become one row in `knowledge/glossary.md`, kept
alphabetized. The file is created on first use with exactly this skeleton (the
YAML `title:` is required — `gen-kb-index.R` dies without it):

```markdown
---
title: "Glossary"
---

Atomic terms the field uses, each with a plain-words definition and a citation.
Terms that accumulate structure get promoted to a page in
[`concepts/`](concepts/) — see `.claude/course-authoring/knowledge-base.md`.

| Term | Plain-words definition | Citation | Related concepts |
|---|---|---|---|
| iid | Independent and identically distributed: each observation is drawn from the same distribution and tells you nothing about the others. | [@boyd2018vmls, §1.2] | [distributions](concepts/distributions.md) |
```

**Promotion rule:** when a glossary term accumulates weight — a second source
disagrees with the definition, misconceptions show up, a derivation is worth
keeping — promote it to its own concept page and replace the row's definition
with a link to that page. Demotion never happens automatically.

## Writing rules

- **One topic per concept page** — the frontmatter carries
  `topic: <slug from .claude/course-authoring/topics.dcf>` (the controlled
  vocabulary shared with foundations' `meta.dcf` `Topic:` field). Topic-grouped
  indexes and enforcement land with issue #125; write the line now on every new
  page. Glossary and source records do NOT get topics (a source spans many; a
  glossary is alphabetical by design).
- **Paper-agnostic prose** — the same rule as foundations. The *page text* is
  neutral field knowledge; which source said what lives in the citations and, when
  they conflict, in the "Source disagreements" section.
- **Every claim cited** — each definition, result, notation variant, misconception,
  and teaching insight carries a bracketed citation with the tightest verified
  pinpoint (`citations.md`). Connective statements that are genuinely yours (a
  bridge sentence, a repo-specific observation) are marked `*(synthesis)*` — use
  sparingly; a page that is mostly synthesis is not knowledge extraction.
- **Learner-aware but not learner-limited** — misconception and teaching-insight
  sections are calibrated to `learner-profile.md`, but definitions and results
  record what the field says, at the field's level.
- **Notation** — notation tables use the `notation-style.md` column format
  (`Symbol | How to say it | What it means`); variants across sources each get a
  cited row, and note which variant this repo's lessons use when one is preferred.

## Merge protocol (second source hits an existing concept)

1. **Add, never overwrite** a cited claim. New support for an existing claim →
   append its citation to the existing bracket.
2. **New notation variant** → new cited row in the Notation table.
3. **New misconception / insight / result** → append, cited.
4. **Genuine conflict** (sources disagree on substance, not phrasing) → do NOT
   pick a winner in the prose. Move the contested claim into "Source
   disagreements" with both positions, both cited. If one position is standard
   and one is fringe, say so — with a citation for why.
5. **Prerequisite edges** — union across sources; an edge only one source implies
   is still an edge.

When several extractors run in parallel (batch extraction), the same protocol
applies plus the check-before-create rule in `.claude/agents/knowledge-extractor.md`
§Parallel/batch operation: re-check the file on disk immediately before creating
any page or row — a sibling agent may have created it since you planned; existing
means merge, never skip or overwrite.

## What the KB is NOT

- Not course content — lessons teach; the KB records. Nothing under `knowledge/`
  follows TEACHING.md's lesson spine.
- Not a copy of the sources — paraphrase and cite; verbatim quoting follows the
  license verdict in the source record (`source-licensing.md`).
- Not learner progress — no Status columns; the KB index is fully machine-owned.
